### Phase B — Profile token-name resolution (fixes blank names on profile lists)
- Root cause (same as the 2026-07-05 "??? Unknown Token" tab title bug): the Envio
  indexer intentionally stores empty `name`/`symbol` strings for every `Token`
  entity. The token detail page already works around this via `useTokenMeta()` in
  `frontend/src/lib/hooks/useData.ts` (on-chain multicall fallback). The profile
  page's "Holdings" and "Tokens Created" lists were NOT doing this fallback — they
  rendered the raw indexer values, which were often blank.
- Fix in `frontend/src/app/profile/[address]/page.tsx`:
  - Imported `useTokensMeta` (existing batch resolver in `useData.ts`).
  - For "Tokens Created" (`tokens` from `useTokensByCreator`, keyed by `id`):
    passed directly through `useTokensMeta` → `tokensResolved`.
  - For "Holdings" (`Holding[]` from `useTokenHoldings`, keyed by `tokenId` not
    `id`): added a local adapter that maps to `{id, name, symbol}[]`, resolves
    via `useTokensMeta`, then merges resolved name/symbol back onto the holdings
    array → `holdingsResolved`.
  - Passed `tokensResolved` to `<CreatedTokensList>` and `holdingsResolved` to
    `<HoldingsList>`.
  - Added `useMemo` to imports.
- No shared-hook signature changes — resolution stays page-level, consistent with
  the token detail page's existing pattern.

### Phase C — Token page Holders list additions
- Extended `frontend/src/lib/hooks/useTokenHolders.ts`:
  - Added `DEAD_ADDRESS` constant = `0x000000000000000000000000000000000000dead`
    (matches `VaultBuybackBurnV2.BURN_ADDRESS` in contracts/src/VaultBuybackBurnV2.sol
    and `LickPair.DEAD` in contracts/src/LickPair.sol — `0xdead` left-pads to the
    same 20-byte address).
  - Added `curveAddress?: string | null` param to the hook signature.
  - Candidate/multicall set now always includes `DEAD_ADDRESS` (unconditional,
    pre- and post-grad) and `curveAddress` (when provided).
  - Extended `TokenHolder` interface with `isCurve?: boolean` and `isBurned?: boolean`
    (mirrors existing `isLp?: boolean`).
  - Tagged results in the build loop with `isCurve`/`isBurned` flags.
  - Updated `useMemo` dependency arrays to include `curveAddress`.
- Updated `frontend/src/app/token/[id]/page.tsx`:
  - Passes `token?.curve ?? null` as the new 7th arg to `useTokenHolders`.
  - Added render branches in the holders table (mirroring the existing `isLp`
    "🌊 Liquidity Pool" blue-tinted row pattern):
    - `holder.isCurve` → 🌊 "Liquidity Pool (unsold supply)" purple-tinted row.
      (User requested the bonding curve row be labeled "Liquidity Pool" for
      consistency with the post-grad DEX pair row — same icon + label, with
      "(unsold supply)" qualifier to distinguish the two states.)
    - `holder.isBurned` → 🔥 "Burned (dead address)" red-tinted row.
- LP row (post-graduation DEX pair) was already implemented and confirmed working
  — no change needed there.
- Burned balance is included in % of total supply (NOT excluded) — gives users a
  true "fraction burned" reading. `totalSupply()` reflects OpenZeppelin's fixed 1B
  mint (no real on-chain burn/decrease happens — only `transfer()` to dead address
  is used by `VaultBuybackBurnV2`).
