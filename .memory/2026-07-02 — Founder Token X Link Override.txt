# 2026-07-02 â€” Founder Token X Link Override

# Context
# User asked to change the X link on the Lickfun Founder token page to point
# at the Lickfun X profile (https://x.com/Lickfun__) instead of whatever the
# token's IPFS metadata says. Other token pages should still use the
# IPFS-provided twitter link.

# Why override at all (vs editing IPFS metadata)
# - The Founder token's IPFS metadata lives in a Storj-pinned JSON that the
#   user controls, but they wanted a guaranteed, app-side override so the
#   page never depends on the on-chain/IPFS data being correct.
# - It mirrors the existing pattern on the Founder token page where we
#   already hardcode the dev-wallet burn tx link (see
#   `frontend/src/app/token/[id]/page.tsx`, "Founder burn notice" block).
# - Single-file change; no other token pages are affected.

# Fix shipped
# File: `frontend/src/app/token/[id]/page.tsx`

# 1. Added a derived `twitterUrl` value next to the existing `isFounderToken`
#    check (which already uses `NEXT_PUBLIC_FOUNDER_TOKEN_ADDRESS`):
#        const twitterUrl = isFounderToken
#          ? "https://x.com/Lickfun__"
#          : ipfsMeta?.twitter;
#
# 2. Updated the "Social links" block in the right-hand "Information" card
#    to use `twitterUrl` instead of `ipfsMeta.twitter` â€” both the outer
#    render condition (`(ipfsMeta?.telegram || twitterUrl || ipfsMeta?.website)`)
#    and the X anchor's `href={twitterUrl}`. The TG and Web buttons are
#    untouched and still driven directly by `ipfsMeta.telegram` / `ipfsMeta.website`.
#
# 3. Follow-up fix (TS narrowing): the original pass also changed
#    `ipfsMeta.telegram` to `ipfsMeta?.telegram` and `ipfsMeta.website` to
#    `ipfsMeta?.website` on the condition + href lines of the TG and Web
#    anchors (i.e. lines that now sit inside a branch whose own condition
#    references `twitterUrl` â€” which can be truthy even when `ipfsMeta` is
#    null). Without the optional chains TypeScript no longer narrowed
#    `ipfsMeta` to non-null inside that block, so bare `ipfsMeta.telegram` /
#    `ipfsMeta.website` accesses failed the type check and broke the
#    Next.js build. The X anchor was already safe (it uses `twitterUrl`).
#    Verified clean with `node node_modules/typescript/bin/tsc -p tsconfig.json --noEmit`.

# Net behaviour
# - Founder token page: X button always opens https://x.com/Lickfun__ in a
#   new tab, regardless of IPFS metadata.
# - Every other token page: X button still uses its own IPFS-provided twitter
#   link (unchanged).
# - If the IPFS metadata has no `twitter` field, the X button is hidden on
#   non-founder pages (same as before). On the Founder page the button is
#   always shown because `twitterUrl` is always truthy.
